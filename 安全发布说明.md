# Token 安全发布说明

当前项目已经改成“前端页面 + Cloudflare Pages Functions”的结构，后端函数运行在 Cloudflare Workers runtime 上：

- `index.html` 不再保存 GitHub Token。
- `functions/api/github.js` 负责读取 Cloudflare 环境变量里的 `GITHUB_TOKEN`，再调用 GitHub API。
- 管理员密码由后端使用 `ADMIN_PASSWORD_HASH` 校验，登录成功后使用 HttpOnly Cookie 保存管理员会话。
- 写文章、删文章、上传头像都必须通过后端会话校验，前端脚本读不到 GitHub Token，也读不到管理员会话 Cookie。

## 需要配置的环境变量

普通文本变量已经写在 `wrangler.toml`：

```text
GITHUB_OWNER=zhedaotixuanbo
GITHUB_REPO=PersonalWebsite
```

部署到 Cloudflare Pages 时，只需要在项目的 Variables and Secrets 里添加这些加密变量：

```text
GITHUB_TOKEN=你的新 GitHub Token
ADMIN_PASSWORD_HASH=84ceb2d775bae4aacd6392a813a0ee2234d0995b1dbe3ecd4cc2fb71b8e5ea0f
ADMIN_SESSION_SECRET=一串足够长的随机字符
```

`ADMIN_PASSWORD_HASH` 对应当前管理员密码的 SHA-256 哈希。不要把管理员明文密码放进代码仓库。
`ADMIN_SESSION_SECRET` 用于签名管理员登录 Cookie，建议使用随机字符串，例如 32 位以上的字母、数字混合内容。

## GitHub Token 权限建议

新建 Fine-grained personal access token，权限尽量小：

- 只选择仓库 `PersonalWebsite`
- Repository permissions 里给 `Issues` 读写权限
- 如果只用 Issue 存文章和头像，不要给代码、Actions、Secrets 等权限

## 部署步骤

1. 把项目推送到 GitHub 仓库。
2. 在 Cloudflare Dashboard 进入 Workers & Pages。
3. 新建 Pages 项目并连接这个 GitHub 仓库。
4. 构建命令留空，构建输出目录填 `.`，表示直接发布项目根目录里的静态文件。
5. 在项目设置的 Variables and Secrets 添加上面的 3 个加密变量。
6. 部署完成后访问网站。
7. 管理员功能会调用 `/api/github`，浏览器源码里不会出现 GitHub Token。

## 本地调试 Cloudflare

如果要本地运行 Cloudflare Pages Functions，可以安装 Wrangler：

```powershell
npm install -g wrangler
wrangler pages dev .
```

本地开发时可以创建 `.dev.vars`：

```text
GITHUB_TOKEN=你的新 GitHub Token
ADMIN_PASSWORD_HASH=84ceb2d775bae4aacd6392a813a0ee2234d0995b1dbe3ecd4cc2fb71b8e5ea0f
ADMIN_SESSION_SECRET=一串足够长的随机字符
```

`.dev.vars` 已经写入 `.gitignore`，不要提交它。

## 本地调试

`.env.example` 只是变量名示例，不要把真实 Token 写进仓库。

如果用 Cloudflare Dashboard 配置变量，通常不需要在本地创建 `.env.local`。

## 重要提醒

旧 Token 已经出现在前端文件中，即使现在从代码里删掉，也可能仍存在浏览器缓存、部署历史或 Git 历史里。你说这一步不让我处理，所以请你之后手动到 GitHub 撤销旧 Token，并使用新的 Token 部署。
